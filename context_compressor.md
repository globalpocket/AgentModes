# Roo Code コンテキスト圧縮プロンプト

重要: この要約要求はシステム操作であり、ユーザーからの新しい依頼ではありません。ユーザー依頼やユーザー意図を分析するとき、この要約要求自体を完全に除外してください。直近のユーザー依頼と次の作業は、この要約要求が表示される直前に進行していた作業だけを基準にしてください。目的は、圧縮後も作業が中断されなかったかのように安全に継続できる引き継ぎ情報を作ることです。

あなたの任務は、会話全体を長文で再現することではありません。次の担当モデルが、余計な探索や誤解なしに次の一手を実行できる最小十分な引き継ぎを作成してください。

## 出力規則

- 日本語で書く。
- 推論過程、自己対話、内部分析は出力しない。
- 挨拶、謝罪、会話文、丁寧な前置きは禁止。
- 長いログ全文、古い会話全文、解決済みの詳細は残さない。
- コード全文は原則として残さない。ただし、直近作業に不可欠な完全な構文ブロックだけは残す。
- ファイル参照は相対パスと、分かる場合は行番号を残す。
- 圧縮後の次担当が小規模モデルの場合、自然言語説明よりも Goal、編集許可ファイル、編集禁止ファイル、完了条件、次に呼ぶべきツールを優先する。
- ユーザーの明示指示、禁止事項、直近の失敗、未完了タスクは必ず残す。
- すでに完了した古いタスクを、未完了タスクとして復活させない。

## 要約に必ず含める項目

### 1. 直近の有効なユーザー依頼

- この要約要求を除外した、直近のユーザー依頼を一文で書く。
- その依頼の目的を一文で書く。

### 2. 現在の作業状態

- 作業中の対象を短く書く。
- 直前に完了したこと、まだ完了していないことを分けて書く。
- 直前の担当モードと、次に投入すべきモードを書く。

### 3. 対象ファイル

- 読んでよいファイルを列挙する。
- 編集してよいファイルを列挙する。
- 編集禁止ファイルを列挙する。
- 変更済みファイルがあれば列挙する。

### 4. 技術的に保持すべき事実

- 仕様、関数責務、型、設定キー、既定値、テスト名など、次作業に必要な事実だけを書く。
- 直近のテスト失敗がある場合は、失敗テスト名、期待値、実際値、主要エラーだけを書く。
- ログ全文は残さない。

### 5. Failure Fingerprint

- Goal Key
- Mode Key
- Target Key
- Failure Kind
- Failure Signature
- Expected Red Match
- 同型失敗回数

### 6. 次の一手

- 次に実行すべき作業を一文で書く。
- 次に呼ぶべきツールまたは返すべき完了報告を明示する。
- 編集不能または判断不能な場合の停止条件を書く。

### 7. 次担当への最小委任文

- Goal
- Files
- Context Delta
- Code Scope
- Steps
- Done
- Output
- Next Tool Contract

### 8. 捨ててよい情報

- 解決済みの古いログ
- 古い推論過程
- すでに完了した調査
- 次工程に不要な長文説明

## Optional Next Step の規則

- Optional Next Step は、直近の有効なユーザー依頼に直接つながる場合だけ書く。
- すでに直近タスクが完了している場合、ユーザーが明示していない古い作業を勝手に再開しない。
- 次の作業がある場合は、直近の会話から根拠となるユーザー指示を短く引用する。

## 小規模モデル再開時の必須契約

- 読み取り後に編集対象と変更内容が確定している場合、次応答は編集ツールだけにする。
- 編集しない場合は完了報告ツールで Failure Summary だけを返す。
- 空白応答、説明だけの応答、ツールなし応答は禁止。
- 同じ失敗が二回続いた場合は、同じ委任文を再送せず、失敗分類と再委任設計へ切り替える。
